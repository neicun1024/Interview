# 这里存放了大数据相关的知识

## Lambda架构
包括批处理层（Batch Layer）、速度层（Speed Layer）、服务层（Serving Layer）；
- 批处理层：存储不可变的原始数据集，定期对全量数据进行重新计算，生成准确的批视图；常用Hadoop、Spark、Hive等；
- 速度层：处理批处理层尚未覆盖的最新增量数据；不存储全量数据，只关注最近一段时间的数据流，生成实时视图；常用Flink、Spark Streaming等；
- 服务层：作为统一查询入口，接收用户请求，同时查询批处理层的批视图和速度层的实时视图，将两者结果合并后返回给用户；常用ClickHouse、HBase、ES、Doris、StarRocks等；

### Lambda架构痛点
1. 两套代码、两套逻辑、两张表；
2. 数据冗余、运维复杂；
3. 结果容易不一致；
4. 改一个需求要改两遍；

## Kapp架构
只有一条实时链路：所有数据进Kafka（保留历史），只用Flink实时计算，结果写入Iceberg/Delta，查询用ClickHouse/Doris/StarRocks等，如果要重算历史，可以重新消费Kafka来刷新Iceberg；

## ETL

- 抽取（Extract）：从异构数据源中识别并复制数据，数据源比如mysql、oracle等；
- 转换（Transform）：将原始数据进行清洗、格式化、计算等操作；
- 加载（Load）：将处理好的数据移入目标存储，比如数据仓库或数据湖；

## Flink/Spark+ClickHouse架构
实时计算+急速查询，是大数据领域最通用的组合。Flink/Spark负责计算，ClickHouse负责存储和查询。所有需要实时计算+海量数据快速查询的场景都能用，比如实时数仓、日志分析、电商/零售实时分析、金融风控等。
Flink负责实时ETL，Spark负责离线ETL，两者协同将处理后的数据存入ClickHouse。

## 流处理引擎
专门设计用于实时或近实时处理连续数据流的数据处理系统。主要应用于：金融欺诈检测、（电商中的）实时推荐、网络监控和异常检测。包括Flink/Spark等。


## 对象存储
以对象为单位管理数据的存储架构，核心是扁平命名空间、全局唯一ID、丰富元数据、HTTP / REST API访问，适合海量非结构化数据；

核心构成：
1. 数据本身：文件、视频、日志等任意内容；
2. 元数据：描述信息，含系统元数据（大小、时间）与自定义元数据（可扩展，用于分类/检索）；
3. 全局唯一ID：扁平地址空间的唯一标识符，替代目录路径，直接定位对象；
关键特性：
1. 扁平结构：无层级目录，访问速度不随数据量增长而下降；
2. 无限扩展：基于分布式架构，可通过增加节点横向扩容；
3. REST API访问：通过HTTP/HTTPS 的 PUT/GET/DELETE 等操作，跨网络与平台兼容；
4. 高可靠：多副本/纠删码 保障数据安全，支持版本控制与生命周期管理；
5. 丰富的元数据支持：用户可自定义元数据（如“项目=电商”、“类型=用户头像”），实现快速检索与智能分类，提升数据可管理性；


## S3（Simple Storage Service，简单存储服务）
一种云上的对象存储服务，S3 API是目前行业通用接口；

核心特点：
1. 存任意非结构化数据：图片、视频、备份、日志、模型文件等；
2. 用KV方式存：没有目录树，只有桶（Bucket）和对象（Object），对象由Data（数据本身）、Metadata（元数据）、Object Key（全局唯一ID）组成；
3. 无限扩容、高可靠、按使用付费；
4. 通过HTTP / REST API访问：PUT/GET/DELETE等；


## Iceberg

### 介绍
Iceberg是跑在S3这类对象存储上的表格式层，它不是存储，是给数据湖加了表结构和事务的一层软件；每次commit都会生成一个快照，每个快照对应一个Manifest list元数据文件。

### 组成
1. Data file（数据文件）：真实数据，比如parquet文件；
2. Manifest file（清单文件）：元数据文件，列出一批Data file的信息，每行对应一个数据文件的详细描述，包括数据文件的状态、文件路径、分区信息、列级别的统计信息（每列的min/max/null count等）、文件的大小、文件里面数据行数等信息；
3. Manifest list（清单列表）：元数据文件，列出一批Manifest file，表示某个快照对应的所有Manifest file，每行对应一个Manifest file的路径、存储的Data file的分区范围、增加/删除了几个数据文件等信息；
4. Metadata json（表的根文件）：指向当前最新的Manifest list，并记录了schema、分区信息、当前表版本等；


### 写入流程
1. 写Data file：生成新的parquet/ORC数据文件，直接写到S3；
2. 写Manifest file：把刚才这批数据文件的路径、大小、分区、统计信息等写到一个新的Manifest file；
3. 写Manifest list：把新的Manifest file和旧快照中有效的Manifest file信息写到一个新的Manifest List；
4. 写Metadata json：生成全新的Metadata json，里面指向新的Manifest list，包含schema、分区信息、当前表版本，文件名带版本号，不覆盖旧文件；
5. 原子提交：只把表的指针指向新的Metadata json，整个写入就原子完成；


### ACID
Iceberg的写入过程，全程只新增文件，不修改、覆盖旧文件，最后一步只换一个指针，成功就是提交，失败就是没提交，旧版本还在，支持时间旅行、回滚。


### 为什么分为Manifest file和 Manifest list
1. 为了在查询时先通过轻量的Manifest list快速过滤分区，只加载需要的Manifest file，大幅减少读操作；
2. 避免每次commit都重写全量清单，commit时只需要新写几个Manifest file，并复用旧的Manifest file来组成Manifest list；



## Parquet
https://waylau.com/about-apache-parquet/

https://zhuanlan.zhihu.com/p/680143641

https://huanglei.rocks/posts/parquet/

一种列式存储文件格式，专门给大数据、数据湖用，存在S3里，被Iceberg管理；
在Parquet文件中，数据每若干行被分做一个Row Group，在同一个Row Group中，不同row的相同列被连续存储在一起；

连续的列每间隔若干行会被分割为一个页（page）；

Row group的核心作用：
1. 读写单元：Parquet读取时，一次IO读一整个Row Group；
2. 压缩效率高：行组中按列存储；
3. 带聚合信息：每列的min/max/null count等，查询时不符合条件的Row Group可以直接跳过；


## 数据湖为什么适合AI
1. 数据量：AI需要海量数据，数据湖用S3/HDFS存储，能满足这个需求；
2. 半/非结构化：AI需要灵活格式，包括结构化、半结构化、非结构化格式，数据湖支持所有数据；
3. Schema灵活：AI特征经常变（增删列或者改结构），数据湖使用Iceberg管理表格式，完美支持；
4. OLAP：AI需要批量、大扫描、列读取，数据湖中数据存储为Parquet文件，用的列存；
5. AI需要增量、版本、可复现，数据湖用Iceberg管理表格式，完美支持；
5. AI需要全生态打通，数据湖是枢纽（一次写入，多处使用）：Flink/Spark洗数据，特征工程、向量化，PyTorch、TensorFlow训练，多人同时用；
总结：AI需要海量、多样、灵活、批量读取、可复现的数据，数据湖完美满足这一切。


## 云器
主要做基于增量计算的云湖仓，采用Kappa架构替代复杂的Lambda架构，实现一套引擎跑批，流，查询。传统的Lambda架构需要两条链路，一条离线链路处理原始表，一条实时链路处理增量实时数据，最后还要将离线数据和实时数据的加工结果进行汇总计算。而云器实现了一体化平台来统一离线和实时链路，使整个架构更加简单可靠。

### 岗位的契合度
1. 湖仓经常使用的ClickHouse、Doris、StarRocks都是LSMTree架构，Iceberg也用了LSMTree里追加写+合并的思想，LSMTree是我比较熟悉的；
2. Parquet中的分为RowGroup，每个RowGroup中都存了每列的聚合信息，类似于我做的skip index；
3. 我主要做的存储查询优化，偏OLAP，非常契合湖仓方向；
4. 现在都在拥抱AI，云湖仓是一个很不错的方向，和AI很契合；
