# 这里存放了数据库的相关知识

## SN和SS
共享存储的计算节点和数据存储完全分离，可以降低存储冗余，快速故障恢复（一个计算节点挂了，其他节点可以直接访问共享存储中的数据，无需等待副本同步或数据迁移），日志服务独立（日志IO不受业务IO干扰，提高整体性能和稳定性）；非共享存储的优点是延迟低性能好，架构简洁，技术成熟，无需依赖外部存储服务，（支持MPP）。

## 共享存储架构
计算层：无状态、可扩缩容；
日志层：高可靠、强顺序、低延迟；
共享存储：存数据、存快照、存归档；

## 查询优化
索引优化（针对where、order by、group by等关键列建索引），SQL写法优化（比如join时候用小表驱动大表），表结构优化（将表进行分区，提升裁剪或者并行处理），架构优化（主从节点，负载均衡）、缓存。

## 物化视图
把复杂查询的结果缓存/落盘，后续查询直接读结果，用于加速复杂/耗时的查询，有数据快照，需要更新。

## OLAP的向量化
向量化是提升OLAP查询性能的常用手段。把指令应用于一个数组或向量，通过CPU数据并行，实现单指令、多数据（简称SIMD）。


## 数据库对比
- MySQL / PostgreSQL：SQL + OLTP + 结构化
- TiDB：NewSQL + OLTP/部分 OLAP 能力 + 结构化/半结构化（取决于用法）
- MongoDB：NoSQL +（偏 OLTP/应用查询）+ 半结构化
- HBase：NoSQL（宽列）+ 典型用于高吞吐随机读写 + 半结构化/稀疏数据
- Elasticsearch：NoSQL + 强项是全文检索 + 半结构化/偏非结构化文本
- ClickHouse / Doris / StarRocks：OLAP + 列式分析 + 结构化/半结构化（常用 JSON/Map）
- Redis：NoSQL（键值）+ 常用于缓存/低延迟读写（可视作 OLTP 场景的一部分）
对象存储 S3/OSS：通常不是数据库，但最适合非结构化“原始文件落库”


## LevelDB/RocksDB
嵌入式KV存储引擎（不是数据库，RocksDB是很多数据库的存储引擎，比如TiDB），提供KV API。使用LSM-Tree架构存储，memtable采用跳表。


## LevelDB和RocksDB的区别
RocksDB在LevelDB的基础上开发，做了很多优化和增强。
1. LevelDB是只支持单线程合并，RocksDB支持多线程合并；
2. LevelDB只有一个Memtable，RocksDB支持多个Memtable；
3. LevelDB只能获取单个KV，RocksDB支持一次获取多个KV，也支持范围查找；
4. LevelDB不支持备份，RocksDB支持全量和增量备份；
5. RocksDB支持列族；


## TiDB
一种分布式NewSQL数据库，支持HTAP。核心组件包括计算层，存储层，调度层。
计算层TiDB Server是无状态的SQL引擎，负责接收、解析、优化和执行sql请求，兼容MySQL协议和语法，支持无缝迁移，可通过增加节点实现横向扩展，适用于高并发OLTP场景；
存储层TiKV是分布式事务型KV存储引擎，基于RocksDB实现持久化，数据按Region分片，使用Multi-Raft协议保证强一致性和高可用，默认三副本部署，支持自动故障转移和数据均衡；
调度层PD（Placement Driver）是集群大脑，负责管理元数据，分配全局时间戳，用于事务一致性，负责Region调度、负载均衡、故障检测与恢复。


## MongoDB
MongoDB是一种NoSQL文档型数据库，以JSON的二进制文档格式BSON存储数据，灵活性高；
特点：水平扩展、高性能、动态schema、丰富的查询语言等；
适用于业务灵活、结构多变的场景，比如内容管理系统、实时分析、物联网、社交媒体等；


## ClickHouse
OLAP列存数据库，主要用于海量数据实时分析。特点包括列存、向量化执行引擎、数据分片。

### ClickHouse向量化执行引擎
大量使用SIMD内置函数对关键代码路径进行优化，通过良好的架构设计和代码设计，使得编译器能够生成良好的向量化代码。

### ClickHouse数据分片
对于Distributed表，会在每个节点建一个local表，用于存放实际数据，而Distributed表本身用于转发数据或者查询，并且收集所有结果，合并为最终的全局结果。

## [StarRocks](https://docs.mirrorship.cn/zh/docs/table_design/table_types/primary_key_table/)

### 集群
集群由FE（FrontEnd）和很多BE（BackEnd）组成；
- FE是前端节点，负责集群管理元数据，管理客户端连接，生成查询计划，查询调度（把查询下发给BE执行）等工作；
- BE是后端节点，负责数据存储，查询计划的执行，compaction，副本管理等工作；每个表的数据根据分区或分桶机制被打散分成多个Tablet，分布在不同的BE上；
每个Tablet内部可以看作一个单机列存引擎，负责对数据的读写操作；

### Tablet
内部分为四个部分：
1. Rowset：一个Tablet的所有数据被切分成多个Rowset，每个Rowset以列存文件的形式存储；
2. Meta（元数据）：保存Tablet的版本历史以及每个版本的信息，会落盘和缓存在内存中；
3. Delete Bitmap：记录每个Rowset中被标记删除的行，会落盘和缓存在内存中；
4. Primary Index（主键索引）：保存主键与该记录所在位置的映射，维护在内存中；

### 写入
数据写入时，对于delete/update操作，会通过主键索引找到数据行原来所在的数据文件以及行号，在delete bitmap中把该行标记为删除；对于update/insert操作，会把最新的数据写入新的数据文件；相当于把update拆成了delete+insert；

### 合并
合并时，会生成新的Rowset和DeleteBitmap（只保留新的数据，不会有多版本行），如果旧的Rowset和DeleteBitmap正在被其他事务访问，不会立刻删除，来实现快照读；
由此可见， StartRocks使用Tablet级版本快照实现读一致性，而非行级MVCC。粒度很大，如果存在长事务，会导致旧版本Rowset和DeleteBitmap都无法清理，导致严重的空间放大；

### 优点
1. 更新性能强：顺序写，高吞吐，适合Kafka实时入仓；
2. 查询性能强：Rowset内部是列存，压缩好，向量化执行，DeleteBitmap支持谓词下推，主键点查、聚合查询都很快；
3. 实现简单、稳定、运维成本低：没有复杂的MVCC版本链，没有undo，compaction逻辑清晰；
4. 唯一性约束
5. 高并发写入不阻塞查询

### 缺点
1. 不支持历史快照查询、不支持闪回；
2. 长事务会导致严重的空间放大（因为旧的Rowset+DeleteBitmap无法GC）；
3. update有写放大（即使只改一列，也要delete+insert整行），不适合高频随机单点更新，更适合批量处理；

## ClickHouse/Doris/StarRocks对比
- ClickHouse：单表查询极快，适合宽表统计；
- Doris：多表join强，易用，运维简单，适合实时数仓；
- StarRocks：综合性能最强、功能最全，支持复杂查询、存算分离与湖仓一体；


## HBase
分布式列存KV数据库，使用LSMTree结构，底层存储基于HDFS实现（冷数据归档、分层存储、离线备份可以放在S3），集群管理基于ZooKeeper实现；

HBase集群由HMaster、RegionServer和ZooKeeper组成；
- HMaster负责管理RegionServer，负责其负载均衡，管理和分配Region，管理namespace和table的元数据，权限控制等；
- RegionServer负责存放和管理本地Region，读写HDFS，管理table中的数据，Client从HMaster中获取元数据，找到Rowkey所在的RegionServer进行读写数据；
- Zookeeper负责存放整个HBase集群的元数据和集群的状态信息，实现HMaster主从节点的故障恢复。

### HBase目录结构
```
hbase/
	data/
		namespace/
			table/
				Region1/
					列族1/
						HFile
						HFile
					列族2/
						HFile
						HFile
	hbaseid/
	WALs/
```

- hbase：HBase在HDFS上的根目录；
- data：数据目录，放所有表的真实数据；
- namespace：命名空间，可以理解为database；
- Region：表按Rowkey范围切成多个Region（横向切分）；
- 列族：同一个Region下，不同列族存放在不同子目录（纵向切分）；每个列族中可以由很多个列，这也体现了HBase的列式存储（列族之间的访问互不影响）；
- HFile：真正的数据文件，内部存了大量行，每行是一个KV，Key=Rowkey+Column Family（列族）+Column Qualifier（列）+Timestamp（用于实现多版本行）+Type（put/delete），V=该单元格的值；
- hbaseid：一个小文件，存了HBase集群唯一标识，主要用于元数据管理、安全认证；
- WALs：放WAL日志的目录；


## ES（ElasticSearch）
分布式搜索与分析引擎，主要用来做全文检索和近实时分析。主要用于在海量数据里按关键词/条件快速检索；对日志/事件/指标做性能分析；支撑搜索推荐、日志平台、可观测性等场景。


## ClickHouse/HBase/ES对比
- ClickHouse：OLAP分析/报表/大屏
- HBase：高并发的点查/宽表
- ES：全文检索/日志检索

三者互补，不是竞争关系
